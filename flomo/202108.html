#inbox 做事的過程 = 計畫、清單、流程

计划

  • 计划，就是把要做的事分解成可操作的步骤。比如你想考英语6级，第一步是买一本6级的书。

  • 你可以看到你的现状跟目标之间的距离，并不断接近。

为什么要做清单？

  • 任务清单，可以让你更清楚完成一件事情的步骤，而自检清单，则可以让你避免遗漏。比如你去火车站，才发现车票忘带了；到了客户的公司，忘了带相应的资料；这些看上去很low，但是不喜欢做自检清单的人一定会经常遇到。

  • 如何做清单？把任务细分成步骤，就变成了清单，而检查这个任务的完成情况，就有了自检清单。

为什么要制定流程？

  • 当我们做的任务清单，在具体操作的过程中，会在不同的位置发现不同的问题。

  • 解决问题点，就是完善流程的过程。

  • 制定流程，是一个刻意练习的过程，就是不断发现问题，解决问题。在新的层面发现新的问题，再解决新的问题。

  • 一旦整个流程没有了问题，就固化成成熟的做事套路，彻底提高效率。

  • 高效的人，就是有了自己的一个个的流程形成的套路。

#project/workflow 用 flomo+excel，搭建自己的工作流

用 flomo+excel，搭建自己的工作流

  • 顆粒度：資訊、任務 的拆解，減少執行推動磨擦力

  • 工程化：建立管道讓自已滑下去，規畫一整套執行流程。

方法論：

  • flomo workday 速记，減少輸入的磨擦力

    • 记录一天工作发生的大小事情，反馈的需求、出现的bug、或者是会议的共识等等。每天一张卡片，标注 day XX，统一在事情发生之后，快速记录在这里。

    • 工作间隙休息时，或者下班前，回看一遍，把信息点再分类流转到不同的excel里面。

      • e.g., 新反馈的需求，进入到「excel_用户story」中。

    • 下班前列出待确认事项，放到第2天的 flomo workday里，作为早上站立会的沟通点。

    • why: 每天零零碎碎的信息很多，不能够让自己老是被打断，靠脑子记忆肯定会遗漏。所以一旦出现新信息，先丢进今天的workday再说，反正下班前会再复一遍（写周报的时候也会）。这样既不会遗漏，也避免对自己的过分打扰。

  • excel workday - 產品/專目整理本週重點

    • 一周5天，作为一个单元。左侧是产品/项目的本周重点（从excel week 的下周计划中复制过来），右侧是落到每天的计划安排。

    • 需要注意的是，无论是预先安排当天工作重点、或者是记录当天核心进度、或者是记录当天发生的重大问题，都必须注意简明扼要，只记录「线索」即可，不要太多细节。因为excel空间不大，写太多文字会丧失可读性。

    • why: 计划下周工作的时候，颗粒度很粗。就是你只知道要做一个事情，但是具体分哪些步骤，是没有写出来的。这样糟糕的地方就是，一是没有办法把控进度，容易黄；二是每天还得想从大的本周工作里面去摘取明天要做什么，又累又焦虑。重要的事情都按照工作日放好位置了，一眼看过去都在，可以给到确定感和安全感。

  • excel weekly 匯總本周的重要事项及进度，也就是周报。

    • 左侧是时间维度，顶部是项目/产品维度。

      • 我喜欢按照M3W2这样的命名方式，表示3月份的第2个礼拜。

      • D表示done，本周完成；P表示plan，下周计划。

      • 当前进行中的版本，同步其本周的进度是否正常、本周重要信息（比如重大变更/问题）。特别重要的用红色标注。

    • why: 因为需要写周报、月报、季度报。写周报时的依据，就是本周的 flomo workday 、excel M5 。最新的周报，放在最上面。这样从上到下浏览的时候，就是本年/本季度/本月的历史周报，不用去翻邮件。excel写完，也可以直接复制到邮件里，不用再重新编辑。

#area/學習 檢驗自已有打通知识的阻塞 - 用test case來檢驗

Test Driven Development (TDD)

  • 把"學習一個知識塊"類比於programming 中的開發一個個的函式功能

    • testing - 要怎麼開發出來的軟體品質呢？通常會寫一些test cases來檢驗

    • test cases - 由高至低可分成 user acceptance, system, integration, unit tests等，對應到的測試對像是 user requirements, system behaviors, sub-system interaction, modules, function 等。又稱v-models

    • automated test execution - test case的執行有分手動執行和自動化執行，能自動化執行的話，測試案例就能低成本而高頻率的不斷重複執行，在每一步做出改變時都用來檢查沒有改壞東西。

    • test coverage - 用來量測"test cases"的檢驗能力，基本常見的是statement coverage 及 branch coverage，即test case execution 是會走完待測程式的式的每一行、每一個if-else的分枝狀況；coverage 越高，表明test quality更有信心。

    • test-driven - 是在unit testing 的層度，先寫好unit tests，才去寫production code。

    • red & green -

  • 學習過程與TDD的概念

    • test first - 先找到新知識(新技能)打算將要怎麼使用，整理下check lists

  • 輸出帶動輸入

  • 在"學習"過程中學到的新概念，有可能會改變或修正舊概念，但自已要怎麼意識到有什麼舊概念是會被改變到呢？

  • 檢查清單(check list)可以做為test cases

#project/卡片盒 基於興趣點的延伸閱讀和思考，輸出成型的觀點

方法論：

  1. 基於興趣的輸入

    • 興趣 = 引起好奇心，能有很多疑問

  2. 盡量記錄想法或心得，不評價好壞

    • 卡片當作最小的知識單位，block

    • 記錄自已的想法，從生活中、文獻、工作專案中的任何念頭

  3. 每日回顧

    • 日知其所無，月無忘其所能

  4. 將卡片產生關聯

    • 新增的卡片要即有卡片產生連結。

    • 量變引起質變

  5. 輸出

    • 寫成文章。依主題輸出成型的觀點

    • 做出決策，就是一種輸出。

    • 解決問題，也是輸出。

    • 会议纪要就是强迫自己组织信息、理解决策产生的好方式。
