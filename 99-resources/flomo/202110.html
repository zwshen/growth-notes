#project/llsw/eng software architecture design

At a minimum, the software architecture design shall provide a blueprint for the development and publication of the software system.  This shall include:

  • Identification of the software components needed to implement, modify, or maintain the software system.

  • Definition of the interfaces between the software components.

    • This should include external interfaces between the software and its operating environment (such as hardware/software interfaces).

  • Analysis of the software architecture, including:

    • Categorization of the software components:

      • Existing software components which will be reused.

      • Existing software components which will be modified.

      • New software components which will be developed or acquired.

    • Identification of responsibility for each software component (collaborate with the PM and/or line management as needed):

      • Which components will be supplied by the project team.

      • Which components will be supplied from outside the project team (a different Garmin project team, purchased/licensed/open source 3rd Party Assets, etc.).

    • Correlation of the software components and interfaces to the software requirements they implement.

For embedded software systems, the Software Architecture Design should also identify anticipated hardware system resource budgets/constraints for each software component.  This should support proactive collaboration between software and hardware teams to ensure the product's design will be adequate to meet customer needs.  Furthermore, this should support making the right design tradeoffs when performing Software Detailed Design and improves the odds that the software system as a whole will stay within budget.

#project/llsw/eng method - software detailed desig

The following steps should be followed to create a software detailed design:

  1. Understand the objectives of the software detailed design:

    1. Identify the version of the software architecture design that the detailed design will derive from.

    2. Identify the software component from the software architecture design that this software detailed design will expand upon.

    3. Identify the software requirements allocated to the software component that this software detailed design must satisfy.

  2. Document the API that will support the interfaces defined in the software architecture.

  3. Document the algorithms and control flow to be implemented in order to satisfy the software requirements allocated to the software component.

    1. This may be done in text (pseudocode, narrative description, etc.), with diagrams (flow charts, state diagrams, etc.), or a combination of both.

    2. The method and extent of the documentation should provide enough detail to show that the allocated software requirements can be met.

      1. Trivial or obvious algorithms may be exempted from documentation.

  4. Document the recommended software solution to implement the design.  

    1. Explore different solutions that could be developed to satisfy the objectives of the software detailed design.

      1. This may include descriptions of classes, functions, or other constructs appropriate for the programming language to be used.

      2. This may include descriptions of recommended software configuration management organization.

    2. Denote any assumptions made for each solution.

    3. Identify the expected resources (CPU, ROM, RAM, etc...) consumed from the operating environment for each solution.

    4. Identify the risks (Technical, Cost/Schedule, Operating Environment, Security, Feasibility) for each solution.

    5. Indicate the desired solution.

      1. If necessary, further detail the chosen solution to minimize ambiguity for implementation.

  5. Document the strategy and/or verification criteria to confirm the implemented software satisfies this design.

  6. Document the estimated effort to perform Software Implementation for this design.

#project/llsw/eng HWM Software Engineers should

Detailed Design Phase Guidelines for Embedded Software Development

Low-Level Software (Hardware/Software Integration) Considerations

  1. Coordinate with Electrical Design Engineers

Throughout the Detailed Design phase, hardware design and verification is dependent upon software support. Software engineers who are tasked with the development of hardware drivers (HWM) should plan their work such that software is available at the time the first hardware prototype boards are available.

The design and development of HWM software requires close collaboration with Electrical Design Engineers. HWM Software Engineers should team with Electrical Design Engineers in planning priorities of all HWM software development and to assist in fully verifying the hardware design in anticipation of the Prototype Design Review. This includes the enabling and exercising of all peripheral technologies such as GPS, sonar, mass data storage and retrieval, etc.

  1. Hardware/Software Integration Support

HWM Software Engineers should also:

  • Support both hardware and software design review activities as needed.

  • Establish a functional Input/Output processing (IOP) module based on the electrical communication technology available in the hardware.

  • Establish support for software diagnostics such as exception reporting; ROM, stack, heap, and other memory consumption statistics; software asserts; etc.

  • Establish fully-functional factory test mode support.

  • Assist other software engineers in integrating high-level applications for operation on target hardware in anticipation of the Prototype Design Review.

#project/卡片盒 IARP 何谓领域：你的长期责任

I.A.R.P 是一组缩写，即：

  • Inbox（收件箱）：我会将所有临时性的，还未消化的内容放置于此，定期来进行归档、整理或者删除。可以当做大脑的缓存，避免记录的时候纠结放在哪里。

  • Area（领域）：这是最重要的概念，即日常你需要精进的「领域」，比如健康就是一个领域，而跑步则是项目；写作是一个领域，而写一篇公众号文章则是一个项目；

  • Resource（资源）：对应到卢曼的 Zettelkasten 方法中，更像是永久笔记。一般来说是兴趣、主题、资产等内容。注意是自己消化过的内容，而非机械的收藏。

  • Project（项目）：是指一个将要发生的独立事件，并且这个事件不是一次性就能完成的，至少需要多个动作才能完成。比如要写一本书，需要整理资料，罗列提纲，撰写内容，联系出版社。类似生活中还有组织一次旅游，录制一期播客等。

#project/卡片盒 效率與主題

要讓卡片盒系統積累的有效率，需不要需限定主題呢？先用專案目標來限定輸入的筆記？是的，要有目標性。

卡片盒的效率不是來自於輸入筆記的數量，而是來自於思考的質量，是來自於對自已過往的經驗、研究、理解的反思結果的產物。

#project/cs/jackson Jackson's Work

  • Telecommunications System Architecture  

    • DFC 方法論

    • I worked for a dozen years (until 2002) with Pamela Zave of AT&T Research on specifying telecommunications systems. This work resulted in the Distributed Feature Composition (DFC) abstract architecture for specifying systems in a modular way that addresses the feature interaction problem effectively. The approach has now been used in commercial products, and is the subject of several patents. Pamela continues to work on deepening and extending the DFC ideas.

  • Information System Development 

    • 稱JSD

    • From the late 1970s to the second half of the 1980s I worked with John Cameron and others on the JSD Method of analysis and design for information systems. The method is based on separating the construction of a model (for example, in a database) of the real world from the construction of the information functions that extract and display the necessary information from the model. The model focuses chiefly on the behaviours of real-world entities over their lifetimes, representing each entity as one or more sequential processes: the local variables of these processes form the data part of the model of the entity. The method is described in my book System Development.

  • Sequential Program Design  

    • 稱JSP

    • From the mid-1960s to the late 1970s I worked with Barry Dwyer, Brian Boulter and other colleagues on the JSP Method of program design. The method bases the program's structure on the structures of its input and output streams. Because the notion of a stream fits any ordered set of messages or records, the method is quite widely applicable. It can be applied, for example, to a sequence of calls of a procedure, and to a sequence of interrupts received by an interrupt handler. Two very effective parts of the method are a systematic way of dealing with structure clashes and a transformation that allows a process to be easily cast in the form either of a 'main program' or of a stateful procedure to produce or consume records of a stream.

    • The method is described in my book Principles of Program Design.
